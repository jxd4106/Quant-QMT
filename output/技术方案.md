## 1. 产品技术画像

### 1.1 产品形态

- 产品类型：**QMT 内置 Python 量化交易策略脚本**（无 GUI、无前端、无后端服务器）
- 运行环境：国金 QMT 桌面客户端（Windows，Python 3.6-3.12，32/64 位）
- 端的数量：0（无用户界面端）——日志和控制台输出是唯一的"界面"
- 主要入口：QMT「模型交易」→ 加载 `half_position_rolling.py` → 实盘模式运行

### 1.2 核心技术约束

| 约束 | 来源（PRD 章节） | 技术含义 |
| :--- | :--- | :--- |
| QMT 内置 Python 环境 | §7 | 只能使用 QMT 自带的 `xtquant` + 标准库 + `numpy`；pip 安装第三方包可能失败或不兼容 |
| handlebar 每约 3 秒触发一次 | §7 | 信号判定不能逐 tick 执行，必须用时间窗口单次触发 |
| ContextInfo 变量跨 tick 回滚 | §7 | 所有持久化状态必须存全局对象或文件，不能存 ContextInfo |
| 策略单线程运行 | §7 | 禁止 time.sleep()、禁止多线程/多进程 |
| 仅 QMT 内置模式（非 miniQMT） | §7 | 历史数据用 `ContextInfo.get_market_data_ex()` 获取，无需额外 subscribe |
| 港股通代码格式为 HGT/SGT | §3 | 行情和交易接口中使用不同后缀（详见 §4.1） |
| A 股 T+1 卖出限制 | §3 | 当日买入股票不能卖出（QMT 自身拦截，策略层暂不做 F11） |
| 港股 T+0 但 T+2 交收 | §8 | 下单前取 `query_stock_asset.cash` 实时可用资金判断 |

## 2. 技术栈总览

```text
┌──────────────────────────────────────────────────────────────┐
│  运行平台   │ 国金 QMT 桌面客户端 (Windows, Python 3.6-3.12)   │
├──────────────────────────────────────────────────────────────┤
│  行情数据   │ xtquant.xtdata  ─ download_history_data        │
│             │                 ─ get_market_data_ex           │
│             │                 ─ get_full_tick                │
├──────────────────────────────────────────────────────────────┤
│  交易执行   │ xtquant.xttrader ─ query_stock_asset           │
│             │                  ─ query_stock_position         │
│             │  passorder()     ─ 限价单下单                   │
├──────────────────────────────────────────────────────────────┤
│  技术指标   │ numpy ─ 纯数组运算（MA/影线/量比/突破判定）       │
├──────────────────────────────────────────────────────────────┤
│  状态持久化 │ Python json ─ state.json（交易状态）             │
│             │              ─ override.json（止损恢复指令）     │
├──────────────────────────────────────────────────────────────┤
│  日志       │ Python logging ─ 按日轮转 FileHandler            │
└──────────────────────────────────────────────────────────────┘
```

### 为什么是这个栈

QMT 内置环境**强约束了技术选型**——只能使用 QMT 自带模块（`xtquant` + `numpy`）+ Python 标准库。不存在"前后端框架选型"——这里的选择不在"用什么"，而在"在约束内怎么组合最高效"。

关键技术决策点（真正可选的）：

1. **技术指标计算**：`numpy` 还是 `pandas`？（§3）
2. **状态持久化格式**：JSON 还是 SQLite？（§4）
3. **日志方案**：Python `logging` 还是 QMT `log_info()`？（§4）

## 3. 核心计算层：技术指标实现

### 3.1 候选方案对比

| 候选 | 内存占用 | 启动速度 | QMT 兼容性 | AI 友好度 | 功能完备度 | 加权总分 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **numpy（纯数组运算）** | 5 | 5 | 5 | 5 | 4 | **4.8** |
| pandas + numpy | 3 | 3 | 2 | 4 | 5 | 3.3 |

权重：QMT 兼容性 40% + 内存/启动 25% + 功能完备 20% + AI 友好 15%。QMT 环境下"能跑起来"是压倒性的硬约束。

### 3.2 推荐：numpy 纯数组运算

**推荐理由**：

1. **QMT 内置 Python 环境中 `pandas` 不一定安装，即使安装了版本也极老。** QMT 自带 `numpy` 是确定可靠的。PRD 调研中多份社区报告提到 pandas 在 QMT 内置 Python 中导入失败或行为异常，降级到 numpy 是唯一稳妥的选择。
2. **内存和启动速度**：numpy 的 import 时间和内存开销远小于 pandas。策略初始化时需要下载和加载历史数据，numpy 的轻量在此处是实际优势。
3. **V7 代码已验证**：现有 V7 版本全部使用 numpy 实现技术指标（`_rolling_mean` 等自定义函数），经过回测验证，不需要重写。
4. **AI 编码友好**：numpy 的 API 稳定、文档丰富，AI 生成 numpy 代码的准确率极高。

**核心配置**：
- 历史数据缓存：`dict[str, dict[str, np.ndarray]]`（股票代码 → {open/high/low/close/volume → array}）
- 60 根日线，日缓存 key = `{stock_code}:{date}`
- 指标计算函数 `calc_indicators()` 接受 5 个 numpy 数组，返回 15 个指标数组

**次选**：pandas —— 只有当 QMT 环境确认安装了 pandas ≥ 1.3 时才考虑。此时 DataFrame 能减少一些手写循环，但信号计算的核心逻辑不需要改。

**不推荐**：纯 Python list 循环 —— 60 根日线 × 15 个指标逐行计算，性能差 10-100 倍，且代码不可读。

### 3.3 风险与缓解

- **风险**：QMT 未来版本更新 `numpy` 大版本，API 不兼容。
- **缓解**：只用 `np.mean` / `np.max` / `np.append` 等基础 API，这些在 numpy 1.x 全程稳定。策略启动时 `print(np.__version__)` 写入日志，方便排查。

## 4. 数据层

### 4.1 行情数据获取

#### 候选方案

| 候选 | 可靠性 | 覆盖面 | 延迟 | QMT 兼容 | 推荐 |
| :--- | :---: | :---: | :---: | :---: | :--- |
| **ContextInfo.get_market_data_ex()** | 5 | 5 | 低 | 5 | ✅ |
| xtdata.get_market_data()（独立模式） | 4 | 5 | 低 | 3 | 次选 |
| xtdata.get_local_data() | 3 | 5 | 极低 | 4 | 不推荐 |

#### 推荐：ContextInfo.get_market_data_ex()

**推荐理由**：
1. **QMT 内置模式的推荐用法**——ContextInfo 由 QMT 框架注入，数据与 QMT 主界面的行情通道同步。
2. V7 代码使用的 `_ctx.get_market_data()` 功能等价，改用 `get_market_data_ex` 可获得更好的字段返回格式。
3. **不选独立 `xtdata.get_market_data()`**——仅 QMT 内置模式，不需要兼容 miniQMT（PRD 已明确不兼容 miniQMT）。

**实时价格获取**：`ContextInfo.get_full_tick([stock_code])`，每次 handlebar 实时调用，返回当前约 3 秒快照的最新价/成交量。**逐股调用（非全市场模式），每次 < 50ms，无 OOM 风险。**

**重要区分——港股代码格式**：

| 接口 | A 股格式 | 港股通格式 |
| :--- | :--- | :--- |
| 行情（get_market_data_ex / get_full_tick） | `603501.SH` | `00700.HK` |
| 交易（passorder / query_stock_position） | `603501.SH` | `00700.HGT`（沪港通）/ `01810.SGT`（深港通） |

策略内部需维护行情代码 → 交易代码的转换映射。

### 4.2 账户与持仓数据

直接从 QMT 交易接口实时查询，不缓存：

```python
# 每次信号判定时实时调用
asset = xt_trader.query_stock_asset(account)
position = xt_trader.query_stock_position(account, stock_code)
```

**不缓存原因**：
- ContextInfo 回滚机制导致缓存会丢失
- 账户数据量极小（单次调用 < 10ms）
- 实时性要求——交易判定必须在最新持仓/资金上做

### 4.3 状态持久化

#### 候选方案

| 候选 | 可读性 | 并发安全 | 容错 | QMT 内置可用 | 推荐 |
| :--- | :---: | :---: | :---: | :---: | :--- |
| **JSON 文件 + 原子写** | 5 | 4 | 4 | 5 | ✅ |
| SQLite | 3 | 5 | 5 | 3 | 次选 |
| pickle 文件 | 1 | 2 | 1 | 5 | 不推荐 |

#### 推荐：JSON + 原子写

**推荐理由**：
1. **用户可读可手动修改**——止损触发后用户可以打开 state.json 确认状态。`override.json` 的"写数组即可恢复"设计直接依赖 JSON 的人类可读性。
2. **Python 标准库即可用，无额外依赖**——QMT 内置 Python 不一定有 `sqlite3` 模块（部分精简版可能移除），但 `json` 一定可用。
3. **QMT 单线程消除了并发写的顾虑**——14:55 单次判定天然串行，不存在两个 handlebar 同时写文件。
4. **原子写保护**：先写 `.tmp` 文件，再 `os.replace()` 覆盖原文件（Windows 下 `os.replace` 是原子操作）。PRD 中 V7 代码已有这个实现模式。

**状态文件结构**：

```python
STATE_SCHEMA = {
    "date": "2026-07-02",
    "stocks": {
        "<code>": {
            "stop_loss_triggered": bool,
            "trade_count_today": int,
            "sold_today": bool,
            "stop_loss_base": float | None
        }
    },
    "daily_stop_count": int
}
```

**override.json 结构**：`["603501.SH", "00700.HGT"]`（一维数组，极简格式，语法出错概率最低）

**次选**：SQLite —— 如果后续需要记录交易历史、统计胜率，SQLite 比 JSON 更适合。但 QMT 内置环境需要先确认 sqlite3 模块可用。

**不推荐**：pickle —— 二进制格式不可读，版本不兼容风险高，安全风险（pickle 反序列化可执行任意代码）。

### 4.4 日志方案

#### 推荐：Python logging + FileHandler

```python
import logging

logger = logging.getLogger('V7')
handler = logging.FileHandler(
    f'logs/V7_{datetime.now():%Y-%m-%d}.log',
    encoding='utf-8'
)
handler.setFormatter(logging.Formatter(
    '%(asctime)s | %(levelname)-7s | %(message)s'
))
logger.addHandler(handler)
```

- 按日轮转（每天一个文件），保留 30 天
- `INFO` 级别及以上同时输出到 QMT 控制台（`StreamHandler`）
- 同类错误 5 分钟内限频 3 条（自定义 `RateLimitedFilter`）

### 5.3 港股 vs A 股下单差异

| 项目 | A 股 | 港股通 |
| :--- | :--- | :--- |
| 市价单 | 支持 | **不支持**——用 `LATEST_PRICE` 模拟 |
| 交易代码后缀 | `.SH` / `.SZ` | `.HGT` / `.SGT` |
| 每手股数 | 固定 100 | 需查 `VolumeMultiple` |
| 最小报价单位 | 0.01 | 查 `PriceTick`（腾讯 0.2 HKD） |
| T+0 买入后当日卖出 | 不可以 | **可以**（但 T+2 资金交收） |

### 5.4 港股手数查询

## 6. 风控实现

### 6.1 15% 止损

```python
def check_stop_loss(stock_code):
    pos = xt_trader.query_stock_position(account, stock_code)
    if pos is None:
        return False
    profit_rate = float(getattr(pos, 'profit_rate', 0) or 0)
    if profit_rate <= -0.15:
        return True  # 触发止损
    return False
```

**关键问题**：`profit_rate` 的语义取决于券商实现——可能是加权平均成本、也可能是最近一次买入成本。PRD 假设 A-004 标注了这一不确定性。技术上的防御措施：

- 策略启动时把 `pos.avg_price` 也记入 `state.stop_loss_base` 作为双重校验
- 止损触发时日志同时打印 `profit_rate` 和基于 `avg_price` 手动计算的浮动盈亏，方便人工复核

### 6.2 全局仓位归一化

```python
total_target = sum(
    cfg['weight'] * TARGET_PCT
    for cfg in STOCK_POOL.values()
)
scale = min(1.0, 1.0 / total_target) if total_target > 0 else 1.0
```

### 6.3 交易时段过滤

```python
def get_active_market(now_time: str) -> str | None:
    """返回 'A' / 'HK_AM' / 'HK_PM' / None"""
    if '09:30' <= now_time <= '11:30' or '13:00' <= now_time <= '15:00':
        return 'A'  # A 股总是有至少一个市场在交易
    # 港股时段判断同时需检查股票池是否含港股

def is_trading_time(stock_code: str, now_time: str) -> bool:
    if stock_code.endswith(('.HGT', '.SGT')):
        return ('09:30' <= now_time <= '12:00') or ('13:00' <= now_time <= '16:00')
    else:
        return ('09:30' <= now_time <= '11:30') or ('13:00' <= now_time <= '15:00')
```

## 7. 运行时架构

### 7.1 QMT 框架生命周期

```text
QMT 启动
  └→ init(ContextInfo)     ——执行一次
       ├── _ctx = ContextInfo
       ├── 遍历 STOCK_POOL: download_history_data()
       ├── 加载 state.json
       ├── 启动对账 (F16)：查券商实际持仓 vs state
       └── 打印初始化完成摘要

策略运行中
  └→ handlebar(ContextInfo) ——每约 3 秒触发一次
       ├── 跨日检测 → 重置每日状态
       ├── 时间检查 → 是否到达信号判定时间点？
       │    ├── 否 → return
       │    └── 是 → 防重复检查 → 全股票池信号判定
       │         ├── 查实时行情 (get_full_tick)
       │         ├── 查持仓/资金 (query_stock_position/asset)
       │         ├── 计算指标 (calc_indicators)
       │         ├── 判定信号 (calc_signals)
       │         ├── 执行卖出 (优先级 S1>S2>S3)
       │         └── 执行买入 (优先级 B1>B2>B3)
       └── 收盘快照 (15:00/16:00) → 写 state.json + 日志摘要
```

### 7.2 全局状态对象

```python
class G:
    """跨 tick 持久化状态。不存 ContextInfo 属性。"""
    trade_count_today: dict[str, int]    # {stock_code: count}
    sold_today: dict[str, bool]           # {stock_code: True}
    stop_loss_triggered: dict[str, bool] # {stock_code: True}
    last_signal_time: str                 # "2026-07-02:14:55"
    daily_stop_count: int                 # 0, 1, 2 → ≥2 阶段二熔断
    last_date: str                        # 跨日检测
    heartbeat_time: str                   # 最后心跳

g = G()
```

**为什么用全局对象而不是闭包/模块变量**：
- ContextInfo 回滚机制——handlebar 每次调用前 QMT 深拷贝 ContextInfo，handlebar 内的 ContextInfo 属性修改会被丢弃
- 全局对象不受此影响，因为 QMT 不会深拷贝整个模块
- 同行确认：迅投官方文档和社区实践一致推荐"自定义全局类对象"方式（`dict.thinktrader.net`）

### 7.3 防重复判定

```python
def should_execute_signals(now_time: str, today: str) -> bool:
    """每日每个判定时间点只执行一次"""
    signal_time = get_signal_time_for_now(now_time)  # '14:55' / '11:55' / '15:55'
    if signal_time is None:
        return False

    key = f'{today}:{signal_time}'
    if g.last_signal_time == key:
        return False  # 已经执行过了

    g.last_signal_time = key
    return True
```

14:55:00 第一次触发 → 执行判定，设置 key。14:55:03 / 14:55:06 … → key 已存在，跳过。

## 8. 实施建议与风险全景

### 8.1 文件结构

```text

├── half_position_rolling.py    # 策略主文件（~600 行）
├── state.json                  # 运行时状态（策略自动维护）
```

### 8.5 全局风险与缓解

| 风险 | 影响 | 缓解措施 |
| :--- | :--- | :--- |
| **QMT 版本更新 API 不兼容** | 策略加载失败或运行崩溃 | `init()` 阶段导入验证所有 API 可用；日志中记录 QMT/xtquant 版本号 |
| **handlebar 在盘中长时间不触发**（QMT 卡死/网络断） | 错过当日信号判定 | 外层 watchdog 脚本检查 state.json 心跳时间戳；超过 5 分钟无更新 → 告警 |
| **passorder 返回成功但实际被拒**（静默失败） | 策略以为下单了但实际没有 | 盘后对账：对比券商实际持仓变化与策略预期下单量；日志完整记录 |
| **港股通代码格式错误**（HGT vs HK） | 查询返回空/下单失败 | `init()` 阶段对 STOCK_POOL 中所有港股通代码做一次 `get_instrument_detail` 预检 |
| **numpy 版本不兼容** | 指标计算报错 | 只用 `np.mean/max/append` 等基础 API（numpy 1.x 全程稳定） |

### 8.6 技术负责人的最终结论

#### 可以立即开工

基于 V7 代码的 numpy 指标计算模块已验证可用，技术方案决策全部收敛，无阻塞性技术未知。

#### 最大的技术不确定性

> **QMT 实盘中的 `profit_rate` 字段语义未经确认。** 止损逻辑的核心输入——`query_stock_position.profit_rate` 是基于加权平均成本还是最新成本？如果基于加权平均成本，多次半仓滚动（高位买低位补）后的 `profit_rate` 可能与直觉偏差较大，导致止损触发时机不准。缓解：策略启动时双重校验——同时用 `profit_rate` 和 `(lastPrice - avg_price) / avg_price` 手动计算，两者交叉验证。

#### 现在最不应该做的事

> **不要引入额外复杂度。** 不加外部配置文件（PRD 已定代码内字典）、不加 miniQMT 兼容、不加多券商适配、不加 Web 监控面板、不加钉钉/微信推送。现阶段唯一的目标是：在国金 QMT 上跑稳 6 信号半仓滚动 + 15% 止损。其余全部后置。
